## ISSUE

**Title:** `[Feature Request]: Independent reproducibility verification for registered evals ‚Äî commit-reveal protocol with tamper-evident HarmonyRecords`

**Post to:** https://github.com/UKGovernmentBEIS/inspect_evals/issues/new

---

EVAL_REGISTER.md's Future Plans describe an upcoming quality checklist that will "quickly communicate to a user which standards and best practices a given eval meets." The existing checks cover whether an eval passes lint and whether an agent can run it. There's one quality dimension none of those checks can address:

> *Has an independent party verified this result, with structural proof that neither party could adjust their verdict after seeing the other's?*

We've built a protocol that provides this ‚Äî and it integrates directly with inspect_evals. This is a proposal to add it as an optional quality dimension in the register.

---

### The problem with self-reported results

When a lab registers an eval result, a user currently has to trust that the number wasn't cherry-picked from multiple runs or adjusted after the fact. A cryptographic attestation bundle (a hash of the run) proves the artifact is internally consistent, but it still asks the reader to trust the party that generated it. What's missing is independent reproducibility ‚Äî a second party tried to get the same number, committed to their finding before seeing the original, and the outcome is permanently recorded somewhere neither party controls.

---

### The solution: ValiChord commit-reveal protocol

[ValiChord](https://github.com/topeuph-ai/ValiChord) is a blind commit-reveal protocol built on Holochain. The mechanism:

1. The original researcher commits a cryptographic hash of their result to a distributed network before any validator starts work
2. Independent validators run the same eval independently and each commit their result blind ‚Äî no party can see another's finding
3. Once all have committed, all reveal simultaneously ‚Äî no last-mover advantage, no post-hoc adjustment possible
4. The outcome is written as a **HarmonyRecord** to the Holochain DHT: permanent, tamper-evident, and held across independent nodes that no single party controls ‚Äî not hosted by inspect_evals and not hosted by us. The exact access mechanism in a production network is still being finalised; the demo illustrates the principle

For inspect_evals specifically, the validators are organisations already running these evals independently ‚Äî other labs, the AISI, academic groups. ValiChord provides the coordination structure that turns existing independent practice into a verifiable, permanent record. The validator type doesn't matter to the protocol: whether a result is reproduced by a human researcher, another lab's automated pipeline, or a scripted re-run, the commit-reveal guarantee is identical.

The most direct example already in the register: **CORE-Bench** evaluates how well an LLM agent computationally reproduces the results of scientific papers. That's precisely the scenario ValiChord is designed for ‚Äî independent parties commit to their reproduction verdict before seeing anyone else's, reveal simultaneously, and the outcome is permanently recorded. A CORE-Bench result with a `valichord_harmony_record_uri` wouldn't just claim reproducibility; it would prove that independent parties reached the same conclusion without being able to coordinate after the fact.

A working demo is live at **[valichord-demo.onrender.com/demo](https://valichord-demo.onrender.com/demo)**. It runs the full commit-reveal protocol and produces a real HarmonyRecord ‚Äî the UI shows a link you can fetch directly to see the raw DHT record, without going through the demo website.

---

### What this looks like in the register

We propose two optional fields in the `evaluation_report` block:

```yaml
evaluation_report:
  commit: <40-char SHA>
  results:
    - model: "anthropic/claude-opus-4-7"
      metrics:
        - {key: accuracy, value: 0.812}
  valichord_attestation_uri: "https://..."        # cryptographic bundle proving this run produced this result
  valichord_harmony_record_uri: "http://..."       # HarmonyRecord on Holochain DHT ‚Äî independently reproducible
```

The **attestation URI** points to a bundle generated by [`valichord_attestation`](https://github.com/topeuph-ai/ValiChord/tree/main/valichord_attestation)'s `InspectEvalsAdapter`. It proves the claimed result matches the run that produced it. Generating one takes five minutes:

```python
from valichord_attestation.adapters import InspectEvalsAdapter

bundle = InspectEvalsAdapter().to_bundle(
    eval_yaml_path="register/my-eval/eval.yaml",
    results_path="results/",
)
# bundle.json ‚Üí host somewhere ‚Üí valichord_attestation_uri
```

The **harmony record URI** is the destination. It points to a HarmonyRecord on the Holochain DHT produced by the full commit-reveal protocol: independent parties each committed to their result before seeing any other, revealed simultaneously, and the outcome was written to a distributed network with no single controlling party. The intent is that the record is independently verifiable by anyone ‚Äî without routing through inspect_evals or through us. The demo shows this working in practice; the exact mechanism in a production network is part of what we're developing.

---

### Why now

EU AI Act Art. 12 requires tamper-evident audit trails with verifiable external evidence for high-risk AI systems. Enforcement begins August 2026. When labs point regulators at inspect_evals results to justify capability claims, a `valichord_harmony_record_uri` in the register entry is the mechanism that makes those claims verifiable by a third party with no stake in the outcome.

---

### Where we are / where we need collaboration

The commit-reveal protocol is built, tested, and running on live Holochain nodes. `valichord_attestation` is production-ready (326 tests, 99% coverage). What takes this from a demo to a real quality signal is the institutional layer: organisations with independent standing running these evals and committing their results. The AISI and its network are exactly that.

This is a collaboration proposal: you bring the eval results worth verifying and the independent parties already running them; we bring the protocol and the infrastructure.

Happy to provide a worked example with a specific register eval, or to discuss what a pilot would look like.

## PR

**Title:** `feat(register): add reproducibility verification fields to EvaluationReport schema`

**Open against:** `UKGovernmentBEIS/inspect_evals` ‚Äî open the issue first and wait for a positive response before opening this.

---

### Summary

Adds two optional fields to `EvaluationReport` in `src/inspect_evals/metadata.py` supporting independent reproducibility verification via the ValiChord commit-reveal protocol. No existing eval.yaml breaks ‚Äî both fields default to `None` and the schema already accepts them silently via `extra="allow"`. Making them typed first-class fields adds validation, doc-rendering, and signals they are a supported quality dimension.

- `valichord_attestation_uri` ‚Äî integrity: proves this run produced the claimed result
- `valichord_harmony_record_uri` ‚Äî reproducibility: permanent HarmonyRecord from the full commit-reveal protocol, on a Holochain DHT no single party controls

### Why `evaluation_report` and not top-level

`EvalMetadataBase` uses `extra="forbid"` ‚Äî top-level fields require schema changes to both the base class and the YAML validator. `EvaluationReport` already uses `extra="allow"` (eval-specific metadata like model provider, run time, notes are all in there), making it the natural home for run-level provenance fields like these. Both attestation and harmony record URIs are properties of a specific run, not of the eval in the abstract ‚Äî which is also why they sit inside `evaluation_report` rather than at the top level.
